# 3.2.3 Diff-Based Telegram Alerts — Implementation

## State File
Location: `/opt/data/state/notify-state.json` (mounted from hermes-data/state/)
Schema: last_check timestamp, state snapshot, last_alert tracking

## Updated Cron Prompt (notification-controller)

```
Read /opt/data/state/notify-state.json to get last known state.
Check current system state: TokenEye health (http://127.0.0.1:8787/__health),
kanban backlog (check board), MEMORY.md size, provider health, cron failures.
Build a state snapshot. Compare against last_state in notify-state.json.

IF state is identical to last known state:
  Increment checks_since_change. Update last_check. Write notify-state.json.
  DO NOT send any Telegram message. (Diff-based suppression.)

IF state changed:
  Reset checks_since_change to 0. Classify changes by tier (INFO/WARNING/CRITICAL).
  Check cooldown: if last_alert is same tier and within cooldown window
  (WARNING=30min, CRITICAL=5min), suppress and just update state.
  Otherwise: compose a single Telegram message with ONLY the changed items.
  Include: what changed, tier, recommended action. Update last_alert.
  Write updated notify-state.json.

Rules:
- Never send "all clear" or "no issues" messages. Silence means healthy.
- Never repeat the same alert within its cooldown window.
- If checks_since_change exceeds 12 (1 hour of no changes), you may skip checks
  for 5 cycles (write "skipped" update to state, resume after).
- Delete stale state entries older than 7 days automatically.
```

## Verification
After deploy, monitor Telegram for 1 hour:
- No duplicate alerts for unchanged state
- State changes trigger single, clean message
- Cooldowns prevent alert storms
